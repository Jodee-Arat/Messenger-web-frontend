# WEB Manual Test Checklist

Составлено на основе:

- `apps/mobile/FULL_TEST_CHECKLIST.txt`
- последних изменений в `apps/web`
- последних багфиксов по DM, модалкам, фото, аватаркам и валидации

Этот файл нужен именно для ручной проверки в web-версии. Здесь перечислено то, что лучше проверить руками в браузере, даже если по коду всё уже выглядит правильно.

## Как использовать

- Идти сверху вниз.
- Отмечать каждый пункт вручную.
- Если что-то ломается, рядом записывать:
  - какой аккаунт использовался
  - какой URL был открыт
  - что нажали перед багом
  - что ожидалось
  - что произошло фактически
  - есть ли ошибка в `Console` / `Network`

## Подготовка перед стартом

- [ ] Поднять `apps/web` в dev-режиме.
- [ ] Открыть приложение в чистой вкладке браузера.
- [ ] Открыть DevTools:
  - вкладка `Console`
  - вкладка `Network`
- [ ] Подготовить минимум 2 аккаунта:
  - основной аккаунт
  - второй аккаунт для DM / сообщений / аватарок / realtime
- [ ] Желательно подготовить 3-й аккаунт для тестов группы и инвайтов.
- [ ] Подготовить данные:
  - хотя бы 1 личный чат
  - хотя бы 1 группа
  - хотя бы 1 обычный чат внутри группы
  - хотя бы 1 сообщение с текстом
  - хотя бы 1 сообщение с фотографией
  - хотя бы 1 сообщение с обычным файлом
- [ ] Если в браузере открыт старый tab после перезапуска dev server, сделать полный reload страницы.

## Блок 1. Smoke Test после входа

- [ ] Открыть главную страницу после логина.
  - Проверить, что нет вечной загрузки.
  - Проверить, что страница не зависает на пустом фоне.
  - Проверить, что sidebar, header и основной контент появились сразу.
  - Ожидаемый результат: приложение открывается без бесконечного loader.

- [ ] Сразу после входа посмотреть `Console`.
  - Ожидаемый результат: нет красных runtime-ошибок React/Next/GraphQL.
  - Примечание: если есть только запросы к `*.js.map`, это отдельно зафиксировать, но главное не путать с runtime-багом UI.

## Блок 2. Авторизация и базовая валидация

### Логин

- [ ] Открыть страницу логина.
- [ ] Ввести корректный логин и пароль.
  - Ожидаемый результат: успешный вход, редирект на приложение.

- [ ] Ввести слишком короткий логин.
  - Ожидаемый результат: форма невалидна, видна ошибка валидации.

- [ ] Ввести пароль без нужного паттерна.
  - Ожидаемый результат: форма невалидна, ошибка показывается до отправки или при попытке отправки.

- [ ] Нажать кнопку показать/скрыть пароль.
  - Ожидаемый результат: пароль корректно переключается между masked/plain text.

### Регистрация

- [ ] Открыть страницу регистрации.
- [ ] Проверить валидный кейс:
  - username длиной 5+ символов
  - корректный email
  - пароль, проходящий паттерн
  - Ожидаемый результат: форма отправляется успешно.

- [ ] Проверить невалидный кейс:
  - короткий username
  - невалидный email
  - слабый пароль
  - Ожидаемый результат: форма не уходит, ошибки отображаются корректно.

## Блок 4. Direct Messages List

Это один из самых важных блоков, потому что здесь недавно были реальные баги.

- [ ] Открыть страницу личных сообщений.
  - Ожидаемый результат: список DM открывается без ошибок.

- [ ] Проверить названия личных чатов.
  - Для каждого DM должно показываться имя собеседника, а не ваше собственное имя.
  - Ожидаемый результат: если вы `heh`, а чат с `kok`, в списке показывается `kok`, а не `heh`.

- [ ] Проверить аватарки в списке DM.
  - Ожидаемый результат: аватар собеседника загружается корректно.
  - Если аватарки нет, должен быть нормальный fallback, а не сломанная картинка.

- [ ] Зайти в конкретный DM и вернуться назад в список.
  - Ожидаемый результат: после возврата имя и аватарка в списке остаются корректными.

- [ ] Проверить превью последнего сообщения.
  - Ожидаемый результат: видно, кто именно написал последнее сообщение.
  - Формат должен быть понятным: автор + текст, либо автор + число файлов.

- [ ] Проверить превью черновика.
  - Ввести черновик в чат, выйти из него, вернуться в список DM.
  - Ожидаемый результат: черновик отображается в превью списка.

- [ ] Проверить время последнего сообщения.
  - Ожидаемый результат: время отображается и не выглядит `NaN` / пустым без причины.

- [ ] Проверить поиск по DM.
  - Ввести часть имени собеседника.
  - Подождать примерно 1.5 секунды.
  - Ожидаемый результат: список фильтруется после debounce, а не мгновенно при каждом символе.

- [ ] Очистить поиск.
  - Ожидаемый результат: полный список возвращается.
